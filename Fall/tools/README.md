
# 🛡️ 智慧長照中心：多軌 AI 異常行為偵測與二審系統

## 📄 專案實作與系統操作說明手冊

---

## 🎯 一、 系統白話文簡介與核心痛點

傳統長照中心依賴護理人員通宵巡房或安裝紅外線感測器，但這存在兩大痛點：**護理人力永遠不足**，以及感測器**誤報率太高**（例如長輩只是半夜翻個身，傳統系統也會大叫，搞到最後護理師身心俱疲，乾脆把警報關掉，反而造成真正的放羊孩子效應）。

本系統打造了一套 **「地端 AI 快速直發 + 雲端/背景大模型二審認證」** 的雙軌數據分流機制。當長輩發生危險時：

1. **地端 AI（第一線偵探）：** 負責秒級反應，一看到高危險動作（跌倒、輪椅意外滑落）且信心度很高，立刻讓護理站閃紅燈，黃金救援不延遲。
2. **多模態大模型（幕後專家）：** 遇到地端 AI 看不清楚、信心度不足的模糊地帶，自動由背景的大語言模型（VLM）接手看圖說故事，判斷是不是誤報，並寫出詳細報告與護理建議。

## 🗺️ 二、 系統架構與「資料的奇幻旅程」

本系統採用非同步微服務架構，資料從攝影機輸入到最終落地，會經歷以下流程：

1. **生產線起點（`inference_test.py`）：** 多路攝影機拉流進來，YOLOv11-Pose 實時抓取骨架，YOLOv11-Seg 抓取不規則物件（如輪椅、床位），並透過自研的 `ActionTransformer` 進行時序行為預測。
2. **交通調度中心（Kafka）：** 邊緣端把資料丟到 Kafka 訊息隊列。信心度極高（>90%）的走「快速道路」直發護理站；需要確認的走「二審慢速道路」送往 VLM。
3. **幕後高材生（`vlm_worker.py`）：** Ollama 帶著 Llava 大模型看實體照片，根據分流 Prompt 用繁體中文寫出巡檢與緊急通報報告。
4. **護理站大螢幕（`monitor_kafka.py`）：** 負責把 Kafka 裡最新、最即時的警報，用紅、黃、藍、紫等高亮工業級顏色渲染在主控台，讓護理師一眼看懂。
5. **中央檔案室（`db_consumer.py` + `db_manager.py`）：** 把所有發生的事件與 VLM 報告，安安靜靜地在背景異步寫入 PostgreSQL 資料庫，不干擾前台運作速度。

## 📁 三、 檔案架構與角色對照表

當評審打開本專案時，各模組功能與檔案角色對照如下：

| 檔案/資料夾名稱              | 它的白話文角色               | 具體軟體工程職責                                                                                             |
| :--------------------------- | :--------------------------- | :----------------------------------------------------------------------------------------------------------- |
| `docker-compose.yml`       | **地基基礎設施**       | 一鍵容器化拉起 Kafka 訊息中樞（9092 埠）與 PostgreSQL 資料庫（5432 埠），免去手動安裝環境的痛苦。            |
| `inference_test.py`        | **第一線 AI 偵探**     | 負責盯著多路攝影機，多執行緒並行拉流，利用 YOLOv11 與 ActionTransformer 抓取長輩異常動作。                   |
| `vlm_worker.py`            | **幕後專家護理長**     | 呼叫本地端 Ollama 運作 Llava 模型。負責看圖做二審審查，並執行定時環境安全巡檢。                              |
| `monitor_kafka.py`         | **護理站警報螢幕**     | 中央監聽服務。實施 MLOps 資料清洗層，將各軌 Payload 對齊轉換，用豐富色彩高亮顯示於終端機畫面上。             |
| `db_consumer.py`           | **中央檔案室書記官**   | 在背景專職偷聽 Kafka 的`processed-reports` 主題，一有新報告就呼叫 Manager 準備寫入資料庫。                 |
| `db_manager.py`            | **資料庫保險箱鑰匙**   | 資料存取物件（DAO）。專門負責管理 PostgreSQL 連線生命週期，並採用參數化查詢徹底防止 SQL 注入攻擊。           |
| `active_learning_dataset/` | **系統自我進化檔案庫** | 當 VLM 發現 YOLO 辨識太弱（盲點）時，自動將該照片與虛擬標註檔（Semi-supervised）打包存於此，供後續模型微調。 |

## 📊 四、 資料庫與 Kafka 資料欄位規格書

### 1. 核心持久化資料表：`ai_processed_reports`

系統每次啟動時，`DatabaseManager` 會執行防禦性建表（`IF NOT EXISTS`），欄位定義如下：

| 欄位名稱 (Column) | 資料型態 (Type)  | 欄位含義 / 白話文解釋         | 技術考量與邏輯                                                              |
| :---------------- | :--------------- | :---------------------------- | :-------------------------------------------------------------------------- |
| `id`            | `SERIAL`       | **自動遞增主鍵**        | 系統內部唯一的流水號，用作資料檢索與排序。                                  |
| `device_id`     | `INT`          | **邊緣運算裝置 ID**     | 標記此警報是由哪一台 Edge 攝影機/設備所觸發。                               |
| `event_type`    | `VARCHAR(100)` | **異常行為事件類型**    | 儲存行為種類（如：`Bed_Exit`, `Wandering`, `Fall`, `chair_slip`）。 |
| `yolo_score`    | `FLOAT`        | **YOLO 邊緣端置信度**   | 記錄地端 YOLOv11 模型在第一時間辨識出該動作的信心分數。                     |
| `vlm_summary`   | `TEXT`         | **VLM 專家審查報告**    | 儲存大語言模型深度分析後的繁體中文白話文現場情境摘要。                      |
| `snapshot_path` | `VARCHAR(255)` | **關鍵影格快照路徑**    | 儲存異常發生時，邊緣端截圖的圖片路徑，便於前端後台調閱圖片。                |
| `detected_at`   | `TIMESTAMP`    | **AI 偵測到事件的時間** | 邊緣端抓到異常的黃金時間（支援 ISO 格式字串自動轉換）。                     |
| `created_at`    | `TIMESTAMP`    | **資料正式落地的時間**  | 記錄這筆資料正式寫入 PostgreSQL 的系統時間，用以計算資料流延遲。            |

### 2. MLOps 中樞欄位對齊與動態相容機制

為了解決前後端非同步開發導致的欄位不一致，`monitor_kafka.py` 在中樞實施了清洗與轉換：

* **警報類型映射：** 將前端原始 `event_type`（"fall", "chair_slip"）自動對齊並提升為工業級的緊急追蹤標籤 `Critical_Fast_Track`。
* **時間標準化：** 自動將 ISO 標準字串（`2026-07-09T11:12:19.456Z`）切片美化為白話的 `YYYY-MM-DD HH:MM:SS` 格式。
* **虛擬唯一主鍵生成：** 當地端快軌直發尚未寫入資料庫、缺乏 ID 時，**利用 Kafka 訊息本身的 `message.offset`（偏移量）**，動態組裝出全域唯一的 `FAST-TRACK-OS-{offset}`，解決前端渲染需要 Unique Key 的難題。

## 🛠️ 五、 系統一鍵啟動與部署 SOP

為了讓使用者能順利跑起整套非同步微服務專案，請開啟 **4 個獨立的終端機（Terminal）視窗**，並按照以下順序執行指令：

### 步驟 1：基礎環境配置與模型載入

請確保已經進入 Python 虛擬環境，並且背景的 Ollama 已經下載好 Llava 大模型。

```bash
# 確保進入專案根目錄 FALL
cd FALL

# 啟動虛擬環境 (Mac/Linux 用第一行；Windows 用第二行)
source .venv/bin/activate
.venv\Scripts\activate

# 確保地端大模型已經下載並在背景待命
ollama run llava:latest
```



### 步驟 2：一鍵啟動 Docker 基礎設施 (容器編排)

利用 Docker Compose 將 Kafka 訊息佇列與 PostgreSQL 資料庫同步在背景架設完成。

```bash
docker compose up -d
```


步驟 3：啟動中央監聽中樞與資料庫落地端 (接收端就位)
在 AI 開始發送警報前，必須先讓護理站接收端與存檔書記官就位。

Bash

# 【請在視窗 A 執行】啟動護理站即時事件通報伺服器 (色彩渲染中樞)

python monitor_kafka.py

# 【請在視窗 B 執行】啟動資料庫落地監聽服務 (異步 PostgreSQL 持久化)

python db_consumer.py



步驟 4：全面發動邊緣推理與 VLM 二審工兵 (發送/處理端運轉)
Bash

# 【請在視窗 C 執行】啟動大模型非同步二審工作節點 (看圖寫繁體中文報告)

python vlm_worker.py

# 【請在視窗 D 執行】啟動邊緣端多鏡頭平行 AI 推理管線 (YOLOv11 骨架/分割拉流)

python inference_test.py



## 💡 六、 專題核心技術亮點（專題口試必勝答辯）

本專案在實作層面具備以下三項業界標準的 MLOps 與軟體工程亮點，可在專題發表時重點突顯：

1. **極致的邊緣端 GPU 算力優化（Inference Cache）：**
   「我們並非盲目運行模型。地端 `YOLO-Pose`（骨架模型）每格執行以求即時捕捉跌倒瞬間；但運算量極大的 `YOLO-Seg`（實例分割）我們做了 **快取複用機制，每 10 幀才計算一次** 。這成功在不犧牲精確度的前提下，幫邊緣設備 **省下了將近 60% 的 GPU 算力壓力** 。」
2. **強健的資料與控制串流分離（Data-Control Separation）：**
   「系統中運算最重的 VLM 大模型模組（`vlm_worker.py`）與資料庫持久化模組（`db_consumer.py`） **完全解耦（Decoupled）** 。VLM 只負責推理並把結果丟回 Kafka，資料庫讀寫則由背景消費端異步消化。這確保了大模型在經歷數秒的圖文推理時， **絕不會卡死護理站的連線或引發資料庫鎖死（Database Lock）** 。」
3. **主動學習自動化閉環（Active Learning Data Loop）：**
   「為了解決長照現場光線、幾何遮擋導致之 AI 盲點，我們設計了主動學習機制。當大模型二審介入時，如果發現地端 YOLO 的信心度落在 0.35 到 0.85 的模糊盲區，`vlm_worker.py` 會 **自動觸發主動學習引擎** ，將該難題相片物理備份，並自動生成 YOLO-Pose 虛擬標註檔（Semi-supervised）， **讓系統具備持續自我進化、越用越聰明的商業價值！** 」
